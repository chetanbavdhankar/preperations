LLM & AI Theory
Deep dive into transformer internals, attention mechanics, inference, and fine-tuning.

Transformer Block Anatomy

Modern LLMs use Pre-LayerNorm (before attention and FFN), unlike the original "post-norm" paper. Pre-norm enables stable training at large scale.

Residual connections: every sub-layer has x = x + sublayer(norm(x)), allowing gradients to flow directly. Without residuals, 100+ layer networks would vanish gradients.
FFN expansion: Feed-forward uses 4× hidden dim. For d_model=4096, FFN hidden dim = 16384. Contains ~⅔ of all parameters.

TransformerBlock (PyTorch)

import torch.nn as nn class TransformerBlock(nn.Module): def __init__(self, d_model, n_heads, ffn_dim, dropout=0.1): super().__init__() self.norm1 = nn.LayerNorm(d_model) self.norm2 = nn.LayerNorm(d_model) self.attn = nn.MultiheadAttention( d_model, n_heads, dropout=dropout, batch_first=True ) self.ffn = nn.Sequential( nn.Linear(d_model, ffn_dim), nn.GELU(), nn.Linear(ffn_dim, d_model), nn.Dropout(dropout) ) self.drop = nn.Dropout(dropout) def forward(self, x, mask=None): # Pre-LayerNorm + residual normed = self.norm1(x) attn_out, _ = self.attn(normed, normed, normed, attn_mask=mask) x = x + self.drop(attn_out) # residual x = x + self.ffn(self.norm2(x)) # residual return x

Method Normalizes Over Used In Key Advantage

class RMSNorm(nn.Module): def __init__(self, dim, eps=1e-6): super().__init__() self.weight = nn.Parameter(torch.ones(dim)) self.eps = eps def forward(self, x): rms = x.norm(2, dim=-1, keepdim=True) * (x.shape[-1] ** -0.5) return self.weight * x / (rms + self.eps)

SwiGLU vs GELU Activation

SwiGLU (used in LLaMA, PaLM): SwiGLU(x,W,V,b,c) = Swish(xW+b) ⊙ (xV+c). Uses 3 weight matrices instead of 2 but achieves better perplexity. The gating mechanism lets the network selectively suppress activations.
GELU (used in GPT-2, BERT): GELU(x) = x·Φ(x) where Φ is the Gaussian CDF. Smooth, non-monotonic, outperforms ReLU on most NLP tasks.

Scaled Dot-Product Attention

Attention(Q, K, V) = softmax( QKᵀ / √d_k ) · V
Why √d_k scaling? Without scaling, the dot products grow as O(d_k), pushing softmax into regions of near-zero gradients (saturation). Dividing by √d_k keeps variance ≈ 1 regardless of dimension.
import torch import torch.nn.functional as F def scaled_dot_product_attention(Q, K, V, mask=None): """Q,K,V: (batch, heads, seq_len, d_k)""" d_k = Q.size(-1) scores = torch.matmul(Q, K.transpose(-2, -1)) / (d_k ** 0.5) if mask is not None: scores = scores.masked_fill(mask == 0, -1e9) attn_weights = F.softmax(scores, dim=-1) return torch.matmul(attn_weights, V), attn_weights

RoPE Implementation

import torch def rotate_half(x): x1, x2 = x[..., :x.shape[-1]//2], x[..., x.shape[-1]//2:] return torch.cat([-x2, x1], dim=-1) def apply_rotary_emb(q, k, cos, sin): """cos, sin: (seq_len, d_k/2) precomputed""" q_rot = (q * cos) + (rotate_half(q) * sin) k_rot = (k * cos) + (rotate_half(k) * sin) return q_rot, k_rot def precompute_rope(seq_len, d_k, base=10000): theta = 1.0 / (base ** (torch.arange(0, d_k, 2).float() / d_k)) t = torch.arange(seq_len) freqs = torch.outer(t, theta) freqs = torch.cat([freqs, freqs], dim=-1) return freqs.cos(), freqs.sin()

Flash Attention

Flash Attention (Dao et al. 2022): Reorders the attention computation to minimize HBM reads/writes. Standard attention materializes the N×N attention matrix to HBM — Flash Attention uses tiled SRAM computation to achieve O(n²) compute but O(n) memory. FA2 is 2-4× faster than standard attention on A100.
Attention Sink phenomenon: The first token (often a newline or BOS) accumulates disproportionately large attention weights regardless of semantic relevance. Theory: softmax must allocate probability mass somewhere — initial tokens become "sinks". StreamingLLM exploits this to enable infinite-length generation by keeping only sink tokens + recent window.

KV Cache Mechanics

KV Cache: During autoregressive generation, each new token needs attention over all previous tokens. Without caching, we'd recompute K,V for all previous tokens every step — O(n²) compute. KV cache stores them once.
Memory = 2 × num_layers × seq_len × d_model × bytes_per_element
Example: LLaMA-70B (80 layers, d=8192, fp16=2 bytes, n=4096) → 2×80×4096×8192×2 ≈ 10.7 GB for just the KV cache.

Batching Strategies

Static Batching: All requests in a batch must finish together. Short sequences wait for the longest one. GPU utilization ~30-40% for mixed workloads.
Continuous Batching: New requests join the batch as soon as a slot opens. Used by vLLM, TGI. GPU utilization 70-90%. 3-10× higher throughput.
PagedAttention (vLLM): Like OS virtual memory for KV cache. Allocates KV blocks non-contiguously, eliminating fragmentation. Enables larger batch sizes and longer sequences. The core innovation behind vLLM's throughput gains.

Speculative Decoding

Speculative Decoding: A small "draft" model (e.g., 7B) generates k tokens quickly, then the large "target" model (70B) verifies all k tokens in a single forward pass. On agreement, accept all k tokens at once. Achieves 2-3× speedup with zero quality loss. Used in production at Google, Meta.

LoRA: Low-Rank Adaptation

ΔW = B · A where B ∈ ℝ^(d×r), A ∈ ℝ^(r×k), r ≪ min(d,k)
LoRA freezes pretrained weights W₀ and injects ΔW = BA into attention layers. Forward pass: h = W₀x + ΔWx = W₀x + BAx. With r=8, d=4096: trains 2×4096×8 = 65,536 vs 4096² = 16.7M original — a ~256× parameter reduction.
QLoRA: Quantize base model to 4-bit NF4 (Normal Float 4 — optimal quantization for normally distributed weights), keep LoRA adapters in fp16. Enables fine-tuning 70B models on a single 48GB GPU. Use double quantization for extra savings.
Rules of thumb: r=8 for style/format tasks (tone, JSON output, templates). r=16-32 for task adaptation. r=64+ for domain knowledge injection (medical, legal). Higher r = more expressivity but more compute.

RAG & Agents
Production RAG architecture, advanced retrieval, vector databases, and agent orchestration.

Naive RAG vs Production RAG

Naive RAG: Query → embed → top-k cosine search → stuff into prompt → generate. Fails with: poor chunking, keyword mismatch, irrelevant retrieved chunks, no evaluation loop.
Production RAG: Query analysis → hybrid retrieval (dense+sparse) → reranking → parent-child expansion → context window management → structured generation → RAGAS evaluation → feedback loop.

Production Pipeline Code

from langchain.text_splitter import RecursiveCharacterTextSplitter from qdrant_client import QdrantClient from qdrant_client.models import Distance, VectorParams, PointStruct from sentence_transformers import SentenceTransformer import uuid # Chunking with RecursiveCharacterTextSplitter splitter = RecursiveCharacterTextSplitter( chunk_size=512, chunk_overlap=102, # ~20% overlap separators=["\n\n", "\n", ". ", " ", ""], length_function=len, ) # Qdrant setup client = QdrantClient("localhost", port=6333) client.create_collection( collection_name="docs", vectors_config=VectorParams(size=1024, distance=Distance.COSINE), ) embed_model = SentenceTransformer("BAAI/bge-large-en-v1.5") def index_documents(docs: list[str]): chunks = splitter.create_documents(docs) texts = [c.page_content for c in chunks] embeddings = embed_model.encode(texts, normalize_embeddings=True) points = [ PointStruct(id=str(uuid.uuid4()), vector=emb.tolist(), payload={"text": text}) for emb, text in zip(embeddings, texts) ] client.upsert("docs", points) def hybrid_retrieve(query: str, top_k=5): # Dense retrieval q_emb = embed_model.encode([query], normalize_embeddings=True)[0] results = client.search("docs", q_emb.tolist(), limit=top_k) return [r.payload["text"] for r in results]

RAGAS Evaluation Metrics

Faithfulness: Are all claims in the answer supported by the retrieved context? (0-1, higher = no hallucination)
Answer Relevancy: How closely does the answer address the original question? (penalizes tangential answers)
Context Precision: What fraction of the retrieved context is actually useful? (penalizes noise in top-k)
Context Recall: Did we retrieve all the information needed to answer? (requires ground truth)

HyDE — Hypothetical Document Embeddings

Problem: A user query "What is the capital of France?" has a very different embedding than a document containing "Paris is the capital and most populous city of France." HyDE closes this gap.
async def hyde_retrieve(query: str, llm, embed_model, vectorstore, k=5): # Step 1: Generate a hypothetical answer (no RAG context) hyde_prompt = f"Write a detailed paragraph that directly answers: {query}" hypothetical_doc = await llm.agenerate(hyde_prompt) # Step 2: Embed the hypothetical document (not the query!) hyde_embedding = embed_model.encode(hypothetical_doc, normalize_embeddings=True) # Step 3: Search with the hypothetical doc's embedding results = vectorstore.similarity_search_by_vector(hyde_embedding, k=k) return results

Self-RAG

Self-RAG (Asai et al. 2023): Trains the LLM to generate special reflection tokens: [Retrieve] (should I retrieve?), [IsREL] (is retrieved chunk relevant?), [IsSUP] (does chunk support generation?), [IsUSE] (is response useful?). The model decides when retrieval helps rather than always retrieving.

Corrective RAG (CRAG)

If retrieval confidence is low, CRAG triggers a web search to supplement or replace the retrieved context. Confidence threshold: if max relevance score < 0.5, fall back to web search. Dramatically reduces hallucination on time-sensitive queries.

HNSW Index Explained

HNSW (Hierarchical Navigable Small World): A multi-layer graph where top layers are sparse (long-range connections) and bottom layers are dense (local connections). Search starts at the top, greedily descends. ef_construction controls index quality (higher = better recall, slower build). m controls graph connectivity (typically 16-64). Achieves ~99% recall at ms latency.
ef_construction: Number of neighbors considered during index build. Higher = better graph quality, slower. Typical: 200.
m: Max connections per node. Higher = better recall, more memory. Typical: 16-32. Memory: O(m × n × 8 bytes).

Retrieval Types

Dense Retrieval: Query and documents embedded to vectors, cosine/dot similarity. Handles paraphrases and semantic similarity. Struggles with exact keyword matches, rare terms, IDs.
Sparse Retrieval (BM25): TF-IDF variant. Excellent for keyword search, exact matches, named entities, codes. Fast, interpretable. Misses semantic similarity.
ColBERT (Late Interaction): Keep all token embeddings per document. At query time: per-query-token max-similarity over all doc tokens, then sum. More expressive than single-vector, much less compute than cross-encoder. Best recall/latency tradeoff.

LangGraph State Machine

from langgraph.graph import StateGraph, END from langgraph.checkpoint.memory import MemorySaver from typing import TypedDict, Annotated import operator class AgentState(TypedDict): messages: Annotated[list, operator.add] retrieved_docs: list[str] answer: str needs_retrieval: bool def analyze_query(state: AgentState) -> AgentState: """Decide if retrieval is needed""" last_msg = state["messages"][-1] # LLM decides whether to retrieve needs_retrieval = True # simplified return {**state, "needs_retrieval": needs_retrieval} def retrieve(state: AgentState) -> AgentState: query = state["messages"][-1] docs = hybrid_retrieve(query) return {**state, "retrieved_docs": docs} def generate(state: AgentState) -> AgentState: context = "\n".join(state["retrieved_docs"]) answer = llm.invoke(f"Context: {context}\nQ: {state['messages'][-1]}") return {**state, "answer": answer} def should_retrieve(state: AgentState) -> str: return "retrieve" if state["needs_retrieval"] else "generate" # Build graph checkpointer = MemorySaver() graph = StateGraph(AgentState) graph.add_node("analyze", analyze_query) graph.add_node("retrieve", retrieve) graph.add_node("generate", generate) graph.set_entry_point("analyze") graph.add_conditional_edges("analyze", should_retrieve, { "retrieve": "retrieve", "generate": "generate" }) graph.add_edge("retrieve", "generate") graph.add_edge("generate", END) app = graph.compile(checkpointer=checkpointer)

Multi-Agent Patterns

Supervisor Pattern: Central LLM routes subtasks to specialized worker agents. Best for: heterogeneous tasks where different agents have different tool access.

02Async LLM Batch Processor
Medium

Process a list of prompts concurrently against the OpenAI API. Implement semaphore rate limiting and exponential backoff on 429 errors.

Time: O(n/concurrency)Space: O(n)
asyncio.Semaphore(N) limits concurrent calls. Catch openai.RateLimitError and sleep with exponential backoff. Use asyncio.gather with return_exceptions=True.
import asyncio, random from openai import AsyncOpenAI, RateLimitError async def batch_llm_process( prompts: list[str], model: str = "gpt-4o-mini", max_concurrent: int = 10, max_retries: int = 4, ) -> list[str | Exception]: client = AsyncOpenAI() sem = asyncio.Semaphore(max_concurrent) async def process_one(prompt: str) -> str: async with sem: for attempt in range(max_retries): try: resp = await client.chat.completions.create( model=model, messages=[{"role": "user", "content": prompt}], timeout=30, ) return resp.choices[0].message.content except RateLimitError: delay = (2 ** attempt) + random.uniform(0, 1) await asyncio.sleep(delay) except Exception as e: if attempt == max_retries - 1: raise await asyncio.sleep(1) raise RuntimeError(f"Max retries exceeded") tasks = [process_one(p) for p in prompts] return await asyncio.gather(*tasks, return_exceptions=True)
03Token Bucket Rate Limiter
Hard

Implement an async token bucket rate limiter class. Must support acquire(tokens=1) that blocks until tokens are available.

Time: O(1) per acquireSpace: O(1)
Track last_refill_time and current_tokens. On acquire: compute elapsed, add (elapsed * rate) tokens up to capacity, then await if insufficient tokens.
import asyncio, time class AsyncTokenBucketRateLimiter: """Token bucket: rate tokens/second, max burst of capacity tokens""" def __init__(self, rate: float, capacity: float): self.rate = rate self.capacity = capacity self.tokens = capacity self._last = time.monotonic() self._lock = asyncio.Lock() async def acquire(self, tokens: float = 1) -> None: async with self._lock: while True: now = time.monotonic() elapsed = now - self._last self.tokens = min(self.capacity, self.tokens + elapsed * self.rate) self._last = now if self.tokens >= tokens: self.tokens -= tokens return wait_time = (tokens - self.tokens) / self.rate await asyncio.sleep(wait_time) # Usage: 10 requests/sec, burst of 20 limiter = AsyncTokenBucketRateLimiter(rate=10, capacity=20) async def make_request(query): await limiter.acquire() return await llm_call(query)

Linear & Logistic Regression

Assumptions (Linear)
  • Linearity between X and y
  • Independence of errors
  • Homoscedasticity (constant variance)
  • Normality of residuals
  • No multicollinearity
L1 (Lasso): Adds |β| penalty → sparse coefficients, feature selection. Use when many irrelevant features.
L2 (Ridge): Adds β² penalty → shrinks all coefficients evenly. Use when all features relevant, high multicollinearity.
When to Use Each
Method When
Linear Reg Continuous target, linear relationship
Logistic Reg Binary/multiclass, interpretable baseline
Lasso p > n, need feature selection
Ridge Correlated features, all relevant
ElasticNet Both concerns apply

Tree Methods Comparison

Method Bias Variance Feature Importance Best For
Decision Tree Low Very High Impurity-based Interpretability, baseline
Random Forest Low-Med Low (bagging) Mean decrease impurity Tabular data, robust baseline
XGBoost Low Low (boosting) Gain-based Kaggle competitions, structured data
LightGBM Low Low Split gain Large datasets, faster than XGBoost
XGBoost uses level-wise tree growth. LightGBM uses leaf-wise growth — faster but needs min_child_samples tuning to prevent overfitting. For >100K rows, LightGBM is typically 3-10× faster.

Clustering Methods

K-Means: Choose K by elbow method (inertia vs K) or silhouette score. Assumes spherical clusters, sensitive to outliers. Scale features first (StandardScaler).
DBSCAN: eps = max distance between neighbors; min_samples = min points in neighborhood. Finds arbitrary shapes, marks outliers as -1. No K needed. Fails in varying density.

Dimensionality Reduction

Method Type Preserves Use When
PCA Linear Global variance Feature reduction before ML, noise removal
t-SNE Non-linear Local structure 2D/3D visualization only (not for ML features)
UMAP Non-linear Local + some global Visualization + downstream ML, faster than t-SNE

Imbalanced Data Strategies

SMOTE: Creates synthetic minority samples by interpolating between neighbors. Works for tabular data, not for high-dimensional sparse data. Don't apply to test set.
class_weight='balanced': Built into sklearn classifiers. Weights loss inversely proportional to class frequency. Simpler than SMOTE, often as effective.
Threshold Tuning: For binary classifiers, optimize threshold by plotting precision-recall curve. Default 0.5 is rarely optimal for imbalanced datasets.

Classification Metrics

Metric Formula Optimize When
Precision TP / (TP + FP) FP cost is high (spam filter — don't block legit email)
Recall TP / (TP + FN) FN cost is high (cancer screening — don't miss disease)
F1 Score 2PR/(P+R) Class imbalance, balanced FP/FN importance
AUC-ROC Area under TPR/FPR curve Binary classification, threshold-independent comparison
AUC-PR Area under P/R curve Highly imbalanced datasets (AUC-ROC misleads)
MCC Balanced metric for all 4 quadrants Severe imbalance, when all errors matter

Cross-Validation Methods

k-fold: Split into k equal folds, train on k-1, test on 1. Repeat k times. Typical k=5 or 10. Assumes IID data.
Stratified k-fold: Maintains class proportions in each fold. Always use for classification with imbalanced classes.
Time-series split: Each fold uses past data to predict future. Never shuffle time-series data — it causes data leakage (training on "future" data).

Statistics Essentials

p-value: Probability of observing data at least as extreme IF H₀ is true. p < 0.05 means <5% chance of this result by random chance. Does NOT measure effect size or practical significance.
Type I Error (α): Reject true H₀ (false positive). Controlled by significance level (typically 0.05).
Type II Error (β): Fail to reject false H₀ (false negative). Reduced by increasing sample size.

Bias-Variance Tradeoff

Total Error = Bias² + Variance + Irreducible Noise
High Bias (Underfitting): Model too simple, misses patterns. Training error high, test error high. Fix: more complex model, more features, less regularization.
High Variance (Overfitting): Model too complex, memorizes noise. Training error low, test error high. Fix: more data, regularization, dropout, simpler model, pruning.

04Missing Data: MCAR vs MAR vs MNAR
Medium

Dataset has missing values in multiple columns. Classify missingness type and apply appropriate imputation strategy for each.

MCAR: random. MAR: missing based on other observed columns. MNAR: missing depends on the missing value itself (e.g., high earners skip salary field).
from sklearn.impute import KNNImputer, SimpleImputer # MCAR → simple mean/median imputation is unbiased simple = SimpleImputer(strategy="median") df["mcar_col"] = simple.fit_transform(df[["mcar_col"]]) # MAR → KNN imputer uses relationships with other columns knn = KNNImputer(n_neighbors=5) df[["mar_col1", "mar_col2"]] = knn.fit_transform(df[["mar_col1", "mar_col2"]]) # MAR → groupby transform for group-aware imputation df["salary"] = df.groupby("department")["salary"].transform( lambda x: x.fillna(x.median()) ) # MNAR → need domain knowledge, or model the missingness itself df["has_salary"] = df["salary"].notna().astype(int) # indicator feature

07Memory-Efficient Chunked CSV Processing
Medium

Process a 50GB CSV file to compute stats and save as Parquet. Must use <2GB RAM.

import pandas as pd from collections import defaultdict dtype_spec = { "user_id": "int32", # int64 → int32 halves memory "score": "float32", # float64 → float32 "category": "category", # string → category codes } agg = defaultdict(list) chunk_iter = pd.read_csv("big_file.csv", dtype=dtype_spec, chunksize=100_000) for i, chunk in enumerate(chunk_iter): chunk_stats = chunk.groupby("category")["score"].agg(["sum", "count"]) for cat, row in chunk_stats.iterrows(): agg[cat].append(row) if i % 10 == 0: print(f"Processed {i*100_000:,} rows") # Combine chunk aggregations final = pd.DataFrame({ cat: {"mean_score": sum(r["sum"] for r in rows) / sum(r["count"] for r in rows)} for cat, rows in agg.items() }).T final.to_parquet("output.parquet", compression="snappy")
