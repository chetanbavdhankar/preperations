🧠 Technical Knowledge Base SENIOR DEPTH

1.1 — Transformer Architecture & Attention Mechanisms

The Transformer (Vaswani et al., 2017) replaced recurrence with self-attention, enabling O(1) path length between any two tokens.

Scaled Dot-Product Attention — Why √dk?

Attention(Q,K,V) = softmax(QKT / √dk) · V

Without the scaling factor, dot products grow with dimensionality, pushing softmax into saturation regions where gradients vanish. Scaling by √dk keeps variance ≈ 1 regardless of head dimension.

Senior insight: In Flash Attention (Dao et al.), the numerically-stable online softmax avoids materializing the full N×N attention matrix — reducing memory from O(N²) to O(N), enabling 128K+ context windows.

Multi-Head Attention — What Do Heads Learn?

Each head projects Q, K, V into a lower-dimensional subspace (dk = dmodel/h). Empirically, different heads specialize: positional attention, syntactic dependency, rare-token focus.

Trade-off: More heads → richer representation but diminishing returns past ~16 heads for d=1024. Grouped Query Attention (GQA, used in Llama 2/3) shares K,V across head groups, cutting KV-cache by 4-8× with negligible quality loss.

Positional Encoding — RoPE vs. ALiBi vs. Sinusoidal
  • Sinusoidal (original): Fixed, cannot extrapolate beyond training length.
  • RoPE (Rotary): Encodes position via rotation matrices in complex space. Naturally supports relative positions. Used in Llama, Mistral, Qwen.
  • ALiBi: No positional embedding — adds linear bias to attention scores. Zero parameters, excellent extrapolation. Used in BLOOM, MPT.

Senior insight: RoPE + NTK-aware scaling allows fine-tuned context extension (e.g., 4K → 32K) without retraining from scratch. This is how Code Llama extended context cheaply.

Decoder-Only vs. Encoder-Decoder — When to Choose What
  • Decoder-only (GPT, Llama): Dominant for generation. Causal mask ensures autoregressive. Simpler to scale (one stack).
  • Encoder-decoder (T5, BART): Superior for seq2seq tasks (translation, summarization) where full bidirectional context on input is valuable.
  • Encoder-only (BERT): Bidirectional, best for classification/NER. Cannot generate fluently.

EPAM relevance: chatbot/Q&A platforms → decoder-only. Document processing pipelines → encoder-decoder may win on extraction quality.

1.2 — RAG Patterns & Vector Databases

Naïve RAG vs. Advanced RAG vs. Modular RAG
  • Naïve RAG: Embed → Retrieve top-K → Stuff into prompt. Fails on multi-hop questions, irrelevant retrieval, and lost-in-the-middle.
  • Advanced RAG: Adds query rewriting, HyDE (hypothetical document embeddings), re-ranking (cross-encoder), and chunk metadata filtering.
  • Modular RAG: Full pipeline with routing (classify query → choose retrieval strategy), iterative retrieval (Self-RAG: retrieve → critique → retrieve again), and tool-augmented retrieval (SQL, API calls, knowledge graphs).

Senior insight: EPAM expects production RAG. That means: evaluation metrics (context precision, faithfulness via RAGAS), guardrails, latency budgets. Most RAG failures are retrieval failures — invest in chunking strategy + re-ranking before tuning the LLM.

Vector DB Trade-offs: Qdrant vs. FAISS vs. Chroma vs. Pinecone
  • FAISS: Facebook's library. Blazing fast (IVF-PQ index), but in-memory only, no metadata filtering natively. Best for batch/offline.
  • Qdrant: Rust-based, production-grade. Supports payload filtering (combine vector + metadata search), scalar/binary quantization. EPAM lists this explicitly.
  • Chroma: Developer-friendly, embedded-first. Good for prototyping, less battle-tested at scale.
  • Pinecone: Fully managed SaaS. Zero-ops but vendor lock-in and cost at scale.

Embedding model choice matters more than DB choice. BGE-M3 (multilingual), text-embedding-3-large (OpenAI), and Cohere embed-v3 each have different dimension/performance trade-offs. Always benchmark on your domain data — general leaderboard rankings mislead.

Chunking Strategies That Actually Matter
  • Fixed-size (e.g., 512 tokens + 50 overlap): Simple, but splits mid-sentence/concept.
  • Semantic chunking: Split on embedding similarity breakpoints. Better coherence, but slower.
  • Recursive character splitting: LangChain default. Hierarchical separators (\n\n → \n → . → " "). Good balance.
  • Document-aware: Use headers/sections as natural boundaries. Best for structured docs (manuals, legal).
  • Parent-child (small-to-big): Retrieve on small chunks for precision, return parent chunk for context. Best of both worlds.

LoRA, QLoRA & Adapter Patterns

LoRA (Low-Rank Adaptation): Freezes base weights, trains low-rank decomposition matrices (A·B) injected into attention layers. Typically rank 8-64. Reduces trainable params by 10,000×.

QLoRA: 4-bit NormalFloat quantized base model + LoRA adapters in BF16. Enables fine-tuning 70B models on a single 48GB GPU.

1.4 — Python "Senior" Concepts

Decorators — Beyond @staticmethod

Decorators are higher-order functions that wrap callables. Senior-level usage:

import functools, time

def retry(max_attempts=3, backoff=2.0):
    """Exponential-backoff retry decorator with jitter."""
    def decorator(func):
        @functools.wraps(func)
        def wrapper(*args, **kwargs):
            for attempt in range(1, max_attempts + 1):
                try:
                    return func(*args, **kwargs)
                except Exception as e:
                    if attempt == max_attempts:
                        raise
                    wait = backoff ** attempt + random.uniform(0, 1)
                    time.sleep(wait)
        return wrapper
    return decorator

@retry(max_attempts=5)
def call_llm_api(prompt: str) -> str: ...

Key: Always use functools.wraps to preserve __name__, __doc__, __module__. Without it, debugging and logging break silently.

Generators & Memory Efficiency
def stream_embeddings(texts: list[str], model, batch_size=32):
    """Yield embeddings in batches — never loads full corpus into memory."""
    for i in range(0, len(texts), batch_size):
        batch = texts[i:i+batch_size]
        yield model.encode(batch)  # returns ndarray, GC'd after consumer processes

Senior insight: Generators + itertools.islice are the backbone of memory-efficient data pipelines. In PySpark, the equivalent is lazy evaluation with .transform() chains — same philosophy, distributed.

Challenge 3 — Missing Data: Intelligent Imputation

Problem: A sensor dataset has ~15% missing values in temperature and humidity. Missing values are not random — sensors fail during extreme weather. Implement a strategy that does NOT use simple fillna(mean).

💡 Hint

Use .interpolate(method='time') for time-indexed data, or .groupby(sensor_id).transform(lambda g: g.fillna(g.rolling(24, min_periods=1).mean())) for sensor-specific rolling imputation.

✅ Solution
# Approach: Per-sensor rolling window imputation (captures local trends)
for col in ['temperature', 'humidity']:
    df[col] = df.groupby('sensor_id')[col].transform(
        lambda s: s.fillna(s.rolling(window=24, min_periods=1, center=True).mean())
    )

# Fallback: any remaining NaNs → forward fill within group
df[['temperature','humidity']] = df.groupby('sensor_id')[['temperature','humidity']].ffill()

# Validate
assert df[['temperature','humidity']].isna().sum().sum() == 0

Why not mean imputation? MNAR (Missing Not At Random) data biases the mean. Rolling window preserves temporal locality. This is the kind of nuance EPAM expects at L5+.

Problem 1 — Build a Minimal RAG Pipeline Medium

Task: Implement a function that takes a list of documents, builds a FAISS index, and answers a query using the top-K retrieved chunks + an LLM call.

Time: O(n·d) build, O(log n) query Space: O(n·d)

✅ Solution
import faiss, numpy as np
from sentence_transformers import SentenceTransformer

class MiniRAG:
    def __init__(self, docs: list[str], model_name="all-MiniLM-L6-v2"):
        self.docs = docs
        self.model = SentenceTransformer(model_name)
        embeddings = self.model.encode(docs, normalize_embeddings=True)
        self.index = faiss.IndexFlatIP(embeddings.shape[1])  # cosine via normalized IP
        self.index.add(embeddings.astype('float32'))

    def query(self, q: str, top_k=3) -> list[str]:
        q_emb = self.model.encode([q], normalize_embeddings=True).astype('float32')
        scores, indices = self.index.search(q_emb, top_k)
        return [self.docs[i] for i in indices[0] if i < len(self.docs)]

# Usage: context = rag.query("What is attention?")
# Then: llm_call(f"Context: {context}\n\nQuestion: {q}")

Problem 2 — Async LLM Batch Processor with Rate Limiting Medium

Task: Process 1,000 prompts through an LLM API with max 20 concurrent requests, exponential backoff on 429s, and result collection.

Time: O(n/concurrency) Space: O(n)

✅ Solution
import asyncio, aiohttp, random

async def process_batch(prompts, api_url, api_key, max_concurrent=20):
    sem = asyncio.Semaphore(max_concurrent)
    results = [None] * len(prompts)

    async def _call(idx, prompt):
        async with sem:
            for attempt in range(5):
                try:
                    async with aiohttp.ClientSession() as s:
                        async with s.post(api_url, json={"prompt": prompt},
                            headers={"Authorization": f"Bearer {api_key}"}
                        ) as r:
                            if r.status == 429:
                                wait = (2 ** attempt) + random.uniform(0, 1)
                                await asyncio.sleep(wait)
                                continue
                            r.raise_for_status()
                            results[idx] = await r.json()
                            return
                except aiohttp.ClientError:
                    if attempt == 4: results[idx] = {"error": str(prompt[:50])}

    await asyncio.gather(*[_call(i, p) for i, p in enumerate(prompts)])
    return results

Problem 3 — Design a Prompt Router Hard

Task: Build a classifier that routes incoming queries to the cheapest capable model: GPT-4o for complex reasoning, GPT-4o-mini for simple Q&A, Claude for creative writing.

Time: O(1) per query Space: O(1)

✅ Solution
from enum import Enum
from pydantic import BaseModel

class ModelTier(str, Enum):
    MINI = "gpt-4o-mini"       # ~$0.15/1M tokens
    FULL = "gpt-4o"            # ~$2.50/1M tokens
    CREATIVE = "claude-sonnet" # best for nuanced writing

class RouteDecision(BaseModel):
    model: ModelTier
    reason: str

ROUTER_PROMPT = """Classify this query into one category:
- SIMPLE: factual lookup, short answer, translation
- COMPLEX: multi-step reasoning, code generation, analysis
- CREATIVE: writing, brainstorming, storytelling

Query: {query}
Respond with just the category name."""

async def route_query(query: str, router_llm) -> ModelTier:
    # Use the cheapest model to classify
    category = await router_llm.complete(
        ROUTER_PROMPT.format(query=query), model="gpt-4o-mini"
    )
    mapping = {"SIMPLE": ModelTier.MINI, "COMPLEX": ModelTier.FULL,
               "CREATIVE": ModelTier.CREATIVE}
    return mapping.get(category.strip().upper(), ModelTier.FULL)

Cost insight: This meta-routing pattern can cut LLM costs by 60-80% in production. The classification call costs fractions of a cent.

from fastapi import FastAPI
from fastapi.responses import StreamingResponse
import openai, json

app = FastAPI()

async def generate_stream(prompt: str):
    client = openai.AsyncOpenAI()
    stream = await client.chat.completions.create(
        model="gpt-4o", stream=True,

        generate_stream(prompt),
        media_type="text/event-stream"
    )

Problem 5 — LLM Output Evaluator (RAGAS-style) Hard

Task: Implement a faithfulness scorer: given a context and an LLM answer, determine what fraction of answer claims are supported by the context.

✅ Solution
DECOMPOSE_PROMPT = """Break this answer into atomic factual claims (one per line):
Answer: {answer}"""

VERIFY_PROMPT = """Given this context, is the following claim supported?
Context: {context}
Claim: {claim}
Reply ONLY 'supported' or 'unsupported'."""

async def faithfulness_score(context: str, answer: str, llm) -> float:
    # Step 1: decompose answer into claims
    claims_raw = await llm.complete(DECOMPOSE_PROMPT.format(answer=answer))
    claims = [c.strip() for c in claims_raw.strip().split('\n') if c.strip()]

    if not claims:
        return 1.0  # no claims = vacuously faithful

    # Step 2: verify each claim against context
    verdicts = await asyncio.gather(*[
        llm.complete(VERIFY_PROMPT.format(context=context, claim=c))
        for c in claims
    ])

    supported = sum(1 for v in verdicts if 'supported' in v.lower())
    return supported / len(claims)  # 0.0 to 1.0

        return "Invalid expression"

    last = state["messages"][-1]

❓ Technical Q&A Bank 25+ QUESTIONS

🟢 Foundational (Warm-up)

Q1: What is the difference between an embedding and a one-hot encoding?

One-hot is sparse, high-dimensional, and captures no semantic similarity (orthogonal vectors). Embeddings are dense, learned representations where cosine similarity encodes semantic relationships. For a 50K token vocab, one-hot = 50K dims; embedding = 768-3072 dims. The embedding space is trained to cluster semantically similar tokens/sentences.

Q2: Explain the difference between cosine similarity and dot product for retrieval.

Cosine similarity normalizes by vector magnitude — it measures angle only. Dot product includes magnitude, so longer vectors score higher. For retrieval: if embeddings are L2-normalized, they're mathematically identical (cos_sim = dot_product). In practice, normalize embeddings and use inner product — it's faster (FAISS IndexFlatIP) and equivalent.

Q3: What is temperature in LLM sampling? What happens at T=0 vs T=2?

Temperature scales logits before softmax: softmax(logits/T). At T→0: argmax (greedy, deterministic). At T=1: standard distribution. At T=2: flattened distribution, more random/creative. For production Q&A systems, T=0-0.3. For creative tasks, T=0.7-1.0. T>1.5 usually degrades coherence.

Q4: What is the context window and why does it matter for cost?

Context window = max tokens (input + output) per call. Cost is proportional to token count. A 128K context GPT-4o call with 100K input tokens costs ~$0.25 just for input. Self-attention is O(n²) in memory — even with Flash Attention's IO optimization, long contexts are expensive. Design principle: minimize context via RAG instead of stuffing everything.

🟡 Intermediate

Q5: How does RAG reduce hallucination? What are its failure modes?

RAG grounds LLM responses in retrieved evidence, reducing fabrication. But failure modes include: (1) Retrieval failure: wrong chunks retrieved → LLM confidently cites irrelevant info. (2) Lost in the middle: LLM ignores information in the middle of long contexts. (3) Conflicting sources: retrieved chunks contradict each other. (4) Over-reliance: LLM parrots context even when its parametric knowledge is more accurate. Mitigations: re-ranking, source-conflict detection, faithfulness scoring.

Q6: Compare LangChain vs. LlamaIndex vs. LangGraph for building agents.

LangChain: General-purpose orchestration. Good for chains and simple agents. Can become "spaghetti" at scale. LlamaIndex: Data-framework-first. Best for RAG pipelines with complex data connectors. Less flexible for non-retrieval agents. LangGraph: State-machine-based agent framework. Explicit graph topology, persistent state, human-in-the-loop. Best for production multi-step agents. EPAM will likely prefer LangGraph for its explicit control flow and debuggability.

Q7: Explain the CAP theorem for vector databases.

Like any distributed system, vector DBs face Consistency-Availability-Partition trade-offs. Qdrant (CP): strongly consistent, may sacrifice availability during network partition. Pinecone (AP): eventually consistent reads, always available. FAISS: single-node, no distribution → CAP doesn't apply. For RAG in production: eventual consistency is usually acceptable — a slightly stale index won't cause errors, just marginally less relevant retrieval.

Q8: What is prompt injection and how do you prevent it?

Prompt injection: user input manipulates the system prompt (e.g., "Ignore previous instructions…"). Defenses: (1) Input sanitization + blocklist. (2) Delimiter-based isolation: wrap user input in XML/markdown tags. (3) Dual-LLM architecture: one LLM classifies intent, another generates response. (4) Output validation: check response against expected schema. (5) Constitutional AI approaches: train models to refuse harmful instructions. No single defense is sufficient — layer them.

Q9: How would you evaluate a chatbot's quality in production?

Multi-dimensional: (1) Automated: RAGAS metrics (faithfulness, answer relevancy, context precision/recall), latency p50/p95/p99, cost per query. (2) Human: Likert-scale user ratings, CSAT, thumbs up/down. (3) Behavioral: conversation completion rate, escalation rate, repeat-question rate. (4) Safety: toxicity detection, PII leakage monitoring. Build a dashboard tracking all four categories. Alert on regression.

Q10: What is the difference between HNSW and IVF indexes in vector search?

HNSW (Hierarchical Navigable Small World): Graph-based. O(log n) query, high recall (~99%+), but memory-hungry (stores full vectors + graph edges). Best for datasets <10M. IVF (Inverted File Index): Clustering-based. Partitions space into Voronoi cells, searches only nearby cells (nprobe parameter). Lower memory with PQ (Product Quantization) compression. Better for 10M+ vectors. Trade-off: HNSW = higher recall, IVF-PQ = lower memory.

🔴 Senior-Level Deep Dives

Q11: You're deploying a RAG system for a bank. Walk me through security considerations.

(1) Data isolation: Multi-tenant vector stores — row-level security or separate collections per tenant. (2) PII in embeddings: Embeddings can leak training data via inversion attacks. Use privacy-preserving embeddings or redact PII before embedding. (3) Prompt injection → data exfiltration: Attacker crafts queries that extract other users' data from context. Defense: strict metadata filtering + output scanning. (4) Audit trail: Log every retrieval + generation for regulatory compliance. (5) Model hosting: On-prem or VPC-deployed models for sensitive data — no external API calls. (6) Output guardrails: NeMo Guardrails or custom validators to block financial advice, PII in responses.

Q12: How do you handle model drift in a production LLM application?

LLMs themselves don't drift (weights are frozen), but the ecosystem drifts: (1) API model updates: OpenAI silently updates models. Pin model versions (e.g., gpt-4o-2024-08-06). (2) Data drift: User queries evolve. Monitor embedding distribution shifts with statistical tests (KS-test on embedding dimensions). (3) Quality drift: Run automated eval suites (golden datasets) on a schedule. Alert when faithfulness/relevancy drops below threshold. (4) Retrieval drift: Knowledge base grows stale. Implement freshness scoring in retrieval ranking.

Q13: Design a multi-agent system for automated code review. What's your architecture?

LangGraph state machine with specialized agents: (1) Planner: Parses diff, identifies changed files, routes to specialists. (2) Security Agent: Scans for vulnerabilities (SQL injection, hardcoded secrets, eval). (3) Style Agent: Checks PEP 8, naming conventions, docstrings. (4) Logic Agent: Analyzes algorithmic correctness, edge cases, complexity. (5) Synthesizer: Aggregates findings, deduplicates, ranks by severity, generates PR comment. State persists between nodes. Human-in-the-loop checkpoint before posting review. Evaluation: measure false positive rate against human reviewers on historical PRs.

Q14: Explain Mixture of Experts (MoE) and why it matters for cost.

MoE activates only a subset of parameters per token (e.g., Mixtral 8x7B: 47B total params, but only 13B active per forward pass). Router network selects top-K experts. Benefits: faster inference (fewer FLOPs), same quality as dense model 3-5x larger. Downsides: higher memory (all experts loaded), load balancing is tricky (expert collapse), harder to fine-tune. Cost implication: Mixtral gives GPT-3.5-level quality at ~3x lower inference cost.

Q15: How would you reduce LLM inference costs by 80% without sacrificing quality?

(1) Prompt routing: Classify query complexity, route simple ones to mini models (see Problem 3 above). (2) Caching: Semantic cache — embed query, check if similar query was answered recently (cosine > 0.95). (3) Prompt compression: LLMLingua or similar to remove redundant tokens from context. (4) Batch processing: Aggregate non-real-time requests. (5) Quantized self-hosted models: Run Mistral-7B-GPTQ for routine tasks, reserve API for complex ones. (6) Output length control: Set max_tokens strictly, use structured output to avoid rambling.

Q16: What's the difference between function calling and tool use in modern LLMs?

Functionally similar but architecturally distinct. Function calling (OpenAI): model outputs a JSON object specifying function name + args. Execution happens client-side. Tool use (Anthropic, LangChain): broader abstraction including APIs, code execution, file operations. The model generates a tool_use block, runtime executes it, returns tool_result. Key insight: neither actually "calls" anything — the model generates structured text that the orchestration layer interprets. Security: always validate tool args server-side, never trust model output as executable code without sandboxing.

Q17: How do you implement guardrails for a production LLM?

Layered approach: (1) Input: PII detection (Presidio), topic classification (block off-topic), prompt injection detection (fine-tuned classifier). (2) During generation: Constrained decoding (LMQL, Outlines) for schema compliance, token-level filtering. (3) Output: Fact-checking against retrieved sources, toxicity scoring, PII scanning on response, schema validation (Pydantic). (4) Operational: Rate limiting per user, cost caps, content moderation API calls, human escalation triggers. Framework options: NVIDIA NeMo Guardrails (programmable rails in Colang), Guardrails AI, or custom FastAPI middleware.

Q18: Explain the attention sink phenomenon and its implications.

In autoregressive transformers, the first token (often BOS/system prompt start) accumulates disproportionately high attention scores across all layers — an "attention sink." This happens because softmax must distribute probability mass somewhere, and the first token becomes a default dump. Implications: (1) Removing the first token crashes generation quality. (2) StreamingLLM exploits this: keep the first few "sink" tokens + a sliding window of recent tokens, enabling infinite-length generation with fixed KV-cache. (3) For RAG: critical context should NOT be placed at the very beginning — it competes with the attention sink.

Q19: You have 1TB of enterprise documents. Design the ingestion pipeline for a RAG system.

(1) Extract: Apache Tika / Unstructured.io for PDF, DOCX, PPTX → markdown. OCR via Tesseract for scanned docs. (2) Clean: Remove headers/footers, tables → markdown format, deduplication (MinHash). (3) Chunk: Semantic chunking with parent-child (retrieve small, return big). Average ~512 tokens. (4) Enrich: Generate title/summary per chunk (LLM), extract entities (NER), add metadata (source, date, access level). (5) Embed: Batch embed with GPU-accelerated model (BGE-M3 for multilingual). (6) Index: Qdrant with HNSW, payload indexes on metadata fields. (7) Orchestrate: Airflow DAG, incremental updates via change-data-capture. (8) Monitor: Track chunk count, embedding drift, query latency, retrieval recall on golden set.

Q20: What are the trade-offs between OpenAI, Anthropic, and open-source models for enterprise?

OpenAI: Best tooling ecosystem (function calling, vision, fine-tuning API). Risk: single vendor, data privacy concerns. Anthropic: Best safety/alignment, excellent for long-context (200K). Constitutional AI approach. More limited tooling. Open-source (Llama, Mistral, Qwen): Full control, on-prem deployment, no data leaves your VPC. Trade-off: you own infra costs, need ML ops expertise, quality lag of ~3-6 months behind frontier. Enterprise recommendation: Multi-provider with abstraction layer (LiteLLM). Use proprietary for quality-critical paths, open-source for high-volume/sensitive data.

Q21: How does KV-cache work and why should you care?

During autoregressive generation, each new token attends to all previous tokens. Without caching, we'd recompute all K,V projections at every step → O(n²) per token. KV-cache stores computed K,V tensors for all previous tokens. Cost: O(n·d·L) memory (n=seq_len, d=head_dim, L=layers). For Llama-70B at 128K context: ~40GB just for KV-cache. Optimization techniques: GQA (share KV across head groups), KV-cache quantization (FP8/INT8), PagedAttention (vLLM — manage KV-cache like virtual memory pages, eliminating fragmentation).

Q22: Explain chain-of-thought prompting and its engineering implications.

CoT prompting (Wei et al.) elicits step-by-step reasoning, improving accuracy on math/logic by 20-40%. Engineering implications: (1) Token cost: CoT increases output length 3-5×. Budget accordingly. (2) Structured CoT: Use XML tags to separate reasoning from final answer → easy parsing. (3) Self-consistency: Generate N reasoning chains, majority-vote on final answer. Expensive but robust. (4) Tree-of-Thought: Branch exploration with backtracking. Most powerful but 10x+ cost. (5) For production: CoT in the reasoning model (o1, Claude thinking), keep the API output clean. Log the reasoning chain for debugging but don't show users.

Q23: What's your approach to testing AI applications?

Four-layer testing pyramid: (1) Unit tests: Mock LLM calls, test parsing, validation, routing logic deterministically. (2) Integration tests: Real LLM calls against golden test set. Assert on structure (JSON schema), not exact text. Use semantic similarity for content assertions. (3) Evaluation suite: RAGAS/DeepEval on 100+ representative queries. Track across deployments. Fail CI if metrics regress >5%. (4) Red-teaming: Adversarial prompt sets (jailbreaks, edge cases, multilingual). Run pre-release. Tools: pytest + deepeval + custom eval harness. Key principle: AI tests are statistical, not deterministic — design for confidence intervals, not exact assertions.

Q24: How would you implement a feedback loop for continuous improvement?

(1) Collect: Thumbs up/down, explicit corrections, implicit signals (conversation length, re-asks, escalations). (2) Analyze: Cluster bad-feedback queries by embedding similarity → identify systematic failure patterns. (3) Fix prioritization: Is it a retrieval failure (bad chunks) → fix chunking/indexing. Knowledge gap → update knowledge base. Format issue → fix prompt template. Hallucination → add guardrail or re-rank. (4) Iterate: A/B test fixes against baseline. Measure lift on the failure cluster. (5) Close loop: Retrained classifiers, updated prompts, expanded golden test set with failure cases.

Q25: Describe the Agentic AI design patterns you've seen work in production.

(1) ReAct (Reasoning + Acting): Think → Act → Observe loop. Works for tool-augmented Q&A. (2) Plan-and-Execute: LLM generates full plan, executor runs steps, re-planner adjusts. Good for multi-step workflows. (3) Multi-agent debate: Multiple LLMs critique each other's outputs. Expensive but high-quality for critical decisions. (4) Supervisor pattern: One agent routes tasks to specialist sub-agents (code, search, analysis). LangGraph excels here. (5) Human-in-the-loop: Agent proposes, human approves at checkpoints. Non-negotiable for production. My Astro Reels pipeline uses a 5-agent LangGraph setup with this exact supervisor pattern.
