LLM Architecture Deep Dive

Theory
Core Architecture
Transformer Block Anatomy

Each transformer layer = Multi-Head Self-Attention (MHA) → Add & LayerNorm → Feed-Forward Network (FFN) → Add & LayerNorm. The FFN typically expands by 4× (d_model → 4*d_model → d_model). This is where ~66% of parameters live in large models.

# Simplified Transformer Layer
class TransformerBlock(nn.Module):
    def __init__(self, d_model, nheads, d_ff, dropout=0.1):
        super().__init__()
        self.attn = nn.MultiheadAttention(d_model, nheads, dropout=dropout)
        self.ff = nn.Sequential(
            nn.Linear(d_model, d_ff),  # d_ff = 4 * d_model
            nn.GELU(),                  # GELU > ReLU for LLMs (smoother gradient)
            nn.Linear(d_ff, d_model)
        )
        self.norm1 = nn.LayerNorm(d_model)
        self.norm2 = nn.LayerNorm(d_model)
        self.dropout = nn.Dropout(dropout)

    def forward(self, x):
        # Pre-LayerNorm (used in GPT-3+, more stable than post-norm)
        attn_out, _ = self.attn(self.norm1(x), self.norm1(x), self.norm1(x))
        x = x + self.dropout(attn_out)          # Residual connection
        x = x + self.dropout(self.ff(self.norm2(x)))
        return x
Modern LLMs (Llama 3, Mistral) use RMSNorm instead of LayerNorm (removes mean-centering, ~10% faster), SwiGLU activation (better than GELU empirically), and Grouped Query Attention (GQA) to reduce KV cache memory.
The Core Equation
Scaled Dot-Product Attention & MHA

Attention(Q,K,V) = softmax(QKᵀ / √d_k) · V — The √d_k scaling prevents vanishing gradients in softmax for large d_k.

import torch
import torch.nn.functional as F

def scaled_dot_product_attention(Q, K, V, mask=None):
    """
    Q, K, V: (batch, heads, seq_len, d_k)
    """
    d_k = Q.shape[-1]
    scores = torch.matmul(Q, K.transpose(-2, -1)) / (d_k ** 0.5)  # (B, H, S, S)
    
    if mask is not None:
        # Causal mask: -inf for future tokens (autoregressive decoding)
        scores = scores.masked_fill(mask == 0, float('-inf'))
    
    weights = F.softmax(scores, dim=-1)   # Attention distribution
    return torch.matmul(weights, V), weights

# Multi-Head: run attention in parallel across H heads
# Each head learns different relationship types:
# Head 1: syntactic dependencies | Head 2: coreference | Head 3: semantic similarity
MHA vs MQA vs GQA: Multi-Head Attention (MHA) — full heads, large KV cache. Multi-Query Attention (MQA) — single K,V shared across heads, 8× less KV memory, slight quality drop. Grouped Query Attention (GQA, used in Llama 2 70B+) — G groups of K,V heads, optimal trade-off. EPAM expects you to know why GQA matters for inference cost.
Flash Attention (Dao et al.) rewrites attention using tiling + recomputation to avoid materializing the full O(n²) attention matrix, achieving O(n) memory. This is what enables 128K+ context windows in production. Name-drop this — it signals you think about deployment, not just training.
Encoding Position
Absolute → Relative → RoPE
# Rotary Position Embedding (RoPE) — used in Llama, Mistral, GPT-NeoX
# Key insight: encodes position in the rotation angle of the query/key vectors
# Benefit: naturally extends to longer sequences than trained on (with scaling tricks)

import torch

def rotate_half(x):
    x1, x2 = x[..., : x.shape[-1]//2], x[..., x.shape[-1]//2:]
    return torch.cat((-x2, x1), dim=-1)

def apply_rotary_emb(q, k, cos, sin):
    # cos/sin are precomputed frequency matrices of shape (seq_len, d_k)
    q_rot = (q * cos) + (rotate_half(q) * sin)
    k_rot = (k * cos) + (rotate_half(k) * sin)
    return q_rot, k_rot

# ALiBi (Press et al.) — linear position bias in attention scores
# Benefit: trained on short sequences, generalizes to longer at inference
# Used in: BLOOM, MPT models
RoPE with YaRN/NTK scaling is how Mistral and Llama 3 extend context at inference. Understanding this at the math level (frequency interpolation) differentiates senior candidates from mid-level ones at EPAM.

KV Cache, Quantization & Batching

Understanding inference bottlenecks is crucial for EPAM's "cost-efficiency" requirement.

# KV Cache: store K, V from all past tokens to avoid recomputation
# Memory: 2 * num_layers * seq_len * d_model * dtype_bytes per request
# GPT-3 175B: ~350GB KV cache at seq_len=4096, fp16 → need batching strategies

# Quantization options for cost reduction:
# INT8 (bitsandbytes): ~2x faster, <1% quality loss on most tasks
# GPTQ (4-bit): ~4x memory reduction, requires calibration dataset
# AWQ (4-bit): activation-aware, better quality than GPTQ
# GGUF (llama.cpp): CPU inference, quantize to 4-8 bit

# Continuous Batching (used in vLLM, TGI):
# Treat each token position as a separate slot
# New requests added mid-generation (vs static batching which waits)
# PagedAttention (vLLM): KV cache as virtual memory pages → 24x throughput gain

# Speculative Decoding:
# Draft model (7B) generates K tokens → target model verifies in parallel
# Achieves 2-3x speedup for the same output quality
vLLM vs TGI vs Ollama: vLLM — highest throughput, PagedAttention, best for production API. TGI (HuggingFace) — broad model support, built-in quantization, easier deployment. Ollama — developer DX, local models, not production-scale. EPAM builds client-facing systems → vLLM or TGI is the right answer.

When to Fine-tune
LoRA / QLoRA / Full FT
  • Consistent output format requirements (JSON schema, structured outputs)

Prompt Engineering Hierarchy

  • GPT-4 / Claude already knows the domain well
  • Interpretability matters — prompt failures are debuggable, FT failures are opaque

import functools, time, asyncio
from typing import Callable, TypeVar

T = TypeVar('T')

# 1. RETRY DECORATOR with exponential backoff (critical for LLM APIs)
def retry_with_backoff(max_retries=3, base_delay=1.0, exceptions=(Exception,)):
    def decorator(func: Callable) -> Callable:
        @functools.wraps(func)  # Preserves __name__, __doc__, __wrapped__
        async def wrapper(*args, **kwargs):
            for attempt in range(max_retries):
                try:
                    return await func(*args, **kwargs)
                except exceptions as e:
                    if attempt == max_retries - 1:
                        raise
                    delay = base_delay * (2 ** attempt)   # Exponential backoff
                    await asyncio.sleep(delay)
        return wrapper
    return decorator

# 2. CACHE DECORATOR for embedding results (avoid re-embedding same text)
def lru_cache_async(maxsize=1024):
    def decorator(func):
        cache = {}
        @functools.wraps(func)
        async def wrapper(*args):
            key = str(args)
            if key not in cache:
                if len(cache) >= maxsize:
                    cache.pop(next(iter(cache)))  # Simple FIFO eviction
                cache[key] = await func(*args)
            return cache[key]
        return wrapper
    return decorator

# 3. TIMER + LOGGING DECORATOR
def timed(logger=None):
    def decorator(func):
        @functools.wraps(func)
        async def wrapper(*args, **kwargs):
            start = time.perf_counter()
            result = await func(*args, **kwargs)
            elapsed = (time.perf_counter() - start) * 1000
            msg = f"{func.__name__} completed in {elapsed:.2f}ms"
            (logger.info(msg) if logger else print(msg))
            return result
        return wrapper
    return decorator

Generators for Large Data Pipelines
# Why generators for AI pipelines?
# Loading 1M document embeddings = OOM. Generators = O(1) memory.

from pathlib import Path
from typing import Iterator, Generator
import json

# Generator for streaming document ingestion
def stream_documents(directory: str) -> Iterator[dict]:
    """Yields documents one at a time — never loads all into memory"""
    for path in Path(directory).rglob("*.json"):
        with open(path) as f:
            for line in f:
                yield json.loads(line)  # JSONL format

# Generator pipeline pattern (composable, lazy)
def chunk_text(docs: Iterator[dict], chunk_size=512) -> Iterator[dict]:
    for doc in docs:
        text = doc["content"]
        for i in range(0, len(text), chunk_size):
            yield {"text": text[i:i+chunk_size], "source": doc["id"]}

def batch_generator(items: Iterator, batch_size=32) -> Iterator[list]:
    """Batch items for efficient API calls (e.g., embedding batches)"""
    batch = []
    for item in items:
        batch.append(item)
        if len(batch) == batch_size:
            yield batch
            batch = []
    if batch:
        yield batch

# Full pipeline (all lazy, O(1) memory)
async def embed_corpus(directory: str):
    docs = stream_documents(directory)          # Generator
    chunks = chunk_text(docs)                   # Generator
    batches = batch_generator(chunks, 32)       # Generator

    async for batch in async_iter(batches):
        embeddings = await embed_batch([c["text"] for c in batch])
        await vector_store.upsert(batch, embeddings)

# Key interview insight: yield from for generator delegation
def flatten_nested(data) -> Generator:
    for item in data:
        if isinstance(item, list):
            yield from flatten_nested(item)  # Recursive delegation
        else:
            yield item

import pandas as pd
from collections import defaultdict

# Efficient chunked processing pattern
def process_large_embedding_metadata(filepath: str, chunksize: int = 50_000):
    cluster_stats = defaultdict(list)
    
    # Specify dtypes upfront — reduces memory by 2-4x
    dtype_spec = {
        'doc_id': 'str',
        'cluster_id': 'int32',    # int32 vs int64 saves 50% for small ints
        'similarity_score': 'float32',  # float32 vs float64 saves 50%
        'word_count': 'int16',
    }
    
    reader = pd.read_csv(
        filepath,
        chunksize=chunksize,
        dtype=dtype_spec,
        usecols=['doc_id','cluster_id','similarity_score','word_count']  # Only needed cols
    )
    
    for i, chunk in enumerate(reader):
        # Process each chunk
        chunk_grouped = (
            chunk.sort_values('similarity_score', ascending=False)
            .groupby('cluster_id')
            .head(10)     # Top 10 per cluster within this chunk
        )
        
        for cluster_id, group in chunk_grouped.groupby('cluster_id'):
            cluster_stats[cluster_id].append(group)
        
        if i % 10 == 0:
            print(f"Processed {(i+1)*chunksize:,} rows...")
    
    # Final merge: combine top-10 candidates per cluster across all chunks
    final_results = {}
    for cluster_id, dfs in cluster_stats.items():
        combined = pd.concat(dfs)
        final_results[cluster_id] = (
            combined.nlargest(10, 'similarity_score')  # Global top-10
        )
    
    return final_results

import asyncio, time

class AsyncTokenBucketRateLimiter:
    """
    Token Bucket: fill rate = max_calls/period, max tokens = max_calls
    Allows bursting up to max_calls, then rate-limits to fill_rate.
    """
    def __init__(self, max_calls: int, period: float = 1.0):
        self.max_calls = max_calls
        self.period = period
        self.tokens = max_calls
        self.fill_rate = max_calls / period
        self.last_fill = time.monotonic()
        self._lock = asyncio.Lock()

    async def acquire(self):
        async with self._lock:
            now = time.monotonic()
            elapsed = now - self.last_fill
            # Refill tokens proportional to elapsed time
            self.tokens = min(self.max_calls, self.tokens + elapsed * self.fill_rate)
            self.last_fill = now

            if self.tokens < 1:
                wait_time = (1 - self.tokens) / self.fill_rate
                await asyncio.sleep(wait_time)
                self.tokens = 0
            else:
                self.tokens -= 1

    async def __aenter__(self):
        await self.acquire()
        return self

    async def __aexit__(self, *args):
        pass

from dataclasses import dataclass, field
from typing import AsyncIterator
import tiktoken

@dataclass
class RetrievedChunk:
    text: str
    source: str
    score: float
    token_count: int = field(init=False)
    
    def __post_init__(self):
        enc = tiktoken.encoding_for_model("gpt-4")
        self.token_count = len(enc.encode(self.text))

class BudgetedRAGChain:
    def __init__(self, retriever, llm_client, max_context_tokens=3000):
        self.retriever = retriever
        self.llm = llm_client
        self.max_context = max_context_tokens
        self.enc = tiktoken.encoding_for_model("gpt-4")

    def _fit_chunks_to_budget(self, chunks: list[RetrievedChunk]) -> list[RetrievedChunk]:
        """Greedy selection: highest-score chunks first, stop when budget exhausted."""
        sorted_chunks = sorted(chunks, key=lambda c: c.score, reverse=True)
        selected, total_tokens = [], 0
        for chunk in sorted_chunks:
            if total_tokens + chunk.token_count <= self.max_context:
                selected.append(chunk)
                total_tokens += chunk.token_count
            else:
                break  # Remaining chunks won't fit (sorted by score, skip all)
        return selected

    async def stream(self, query: str) -> AsyncIterator[str]:
        # Retrieve
        raw_chunks = await self.retriever.aretrieve(query, k=20)
        chunks = [RetrievedChunk(c.text, c.source, c.score) for c in raw_chunks]
        
        # Budget-fit context
        selected = self._fit_chunks_to_budget(chunks)
        context = "\n\n---\n\n".join(c.text for c in selected)
        
        # Log retrieval metrics
        metrics = {
            "retrieved": len(raw_chunks),
            "used": len(selected),
            "context_tokens": sum(c.token_count for c in selected),
            "mean_score": sum(c.score for c in selected) / len(selected) if selected else 0
        }
        await self._log_metrics(query, metrics)
        
        # Stream LLM response
        prompt = f"Context:\n{context}\n\nQuestion: {query}\nAnswer:"
        async for token in self._stream_llm(prompt):
            yield token

🟠 Problem 3: Semantic Cache for LLM Responses
Implement a semantic cache that returns cached LLM responses for semantically similar queries (cosine similarity > threshold), reducing API costs.
Medium-Hard Time: O(n·d) Space: O(n·d) 
import numpy as np
from typing import Optional
import asyncio

class SemanticCache:
    """
    Caches LLM responses by semantic similarity of queries.
    Hit if cosine_similarity(new_query_emb, cached_emb) >= threshold.
    """
    def __init__(self, embedder, threshold: float = 0.95, max_size: int = 1000):
        self.embedder = embedder
        self.threshold = threshold
        self.max_size = max_size
        self.embeddings: list[np.ndarray] = []
        self.responses: list[str] = []
        self.queries: list[str] = []
        self._lock = asyncio.Lock()

    def _cosine_sim(self, a: np.ndarray, b: np.ndarray) -> float:
        return float(np.dot(a, b) / (np.linalg.norm(a) * np.linalg.norm(b) + 1e-8))

    async def get(self, query: str) -> Optional[tuple[str, float]]:
        """Returns (cached_response, similarity) if cache hit, else None."""
        if not self.embeddings:
            return None
        
        query_emb = await self.embedder.aembed(query)
        
        # Vectorized similarity against all cached embeddings
        cache_matrix = np.vstack(self.embeddings)  # (n, d)
        sims = cache_matrix @ query_emb / (
            np.linalg.norm(cache_matrix, axis=1) * np.linalg.norm(query_emb) + 1e-8
        )
        
        best_idx = np.argmax(sims)
        best_sim = float(sims[best_idx])
        
        if best_sim >= self.threshold:
            return self.responses[best_idx], best_sim
        return None

    async def set(self, query: str, response: str):
        async with self._lock:
            query_emb = await self.embedder.aembed(query)
            if len(self.embeddings) >= self.max_size:
                self.embeddings.pop(0)  # FIFO eviction
                self.responses.pop(0)
                self.queries.pop(0)
            self.embeddings.append(query_emb)
            self.responses.append(response)
            self.queries.append(query)

🟡 Problem 4: Document Chunking Strategy
Implement a hierarchical chunking function that splits documents into parent (large context) and child (precise retrieval) chunks, maintaining parent-child links.

from dataclasses import dataclass, field
import uuid

@dataclass
class Chunk:
    text: str
    chunk_id: str = field(default_factory=lambda: str(uuid.uuid4())[:8])
    parent_id: str | None = None
    level: str = 'child'   # 'parent' or 'child'
    metadata: dict = field(default_factory=dict)

def hierarchical_chunk(
    document: str,
    parent_size: int = 1024,
    child_size: int = 256,
    overlap: int = 32
) -> list[Chunk]:
    """
    Parent chunks: large context window (retrieved for LLM input)
    Child chunks: small, precise (used for similarity search)
    Relationship: each child knows its parent_id
    """
    def split_with_overlap(text: str, size: int, overlap: int) -> list[str]:
        """Split text into chunks with overlap using word boundaries."""
        words = text.split()
        chunks, i = [], 0
        while i < len(words):
            chunk_words = words[i:i + size]
            chunks.append(' '.join(chunk_words))
            i += size - overlap
        return chunks

    # Approximate word count from token count (1 token ≈ 0.75 words)
    parent_words = int(parent_size * 0.75)
    child_words = int(child_size * 0.75)

    all_chunks = []
    parent_texts = split_with_overlap(document, parent_words, int(parent_words*0.1))

    for parent_text in parent_texts:
        parent = Chunk(text=parent_text, level='parent')
        all_chunks.append(parent)
        
        # Create child chunks from the parent text
        child_texts = split_with_overlap(parent_text, child_words, int(child_words*0.15))
        for child_text in child_texts:
            child = Chunk(
                text=child_text,
                parent_id=parent.chunk_id,
                level='child',
                metadata={'parent_preview': parent_text[:100]}
            )
            all_chunks.append(child)

    return all_chunks

What is the difference between pre-training, fine-tuning, and RLHF? When would you choose each?Senior›
Pre-training: unsupervised learning on massive corpus (predict next token). Creates a base model with broad world knowledge. Cost: millions of GPU-hours. Fine-tuning (SFT): supervised training on instruction-following pairs (prompt→response). Teaches the model format and task compliance. Cost: hours to days. RLHF: trains a reward model from human preference pairs, then uses PPO to optimize the LLM against this reward. Aligns outputs with human values, reduces hallucination. Cost: moderate but complex. DPO replaces RLHF's PPO with a simpler loss — same result, less engineering overhead. EPAM answer: for client projects, always start with prompting → SFT if needed → DPO for alignment. Explain why Transformers use Layer Normalization instead of Batch Normalization.Senior›
Batch Norm normalizes across the batch dimension (mean/std computed over batch), making it dependent on batch size. For NLP, sequences in a batch have variable lengths, and BN would mix statistics across unrelated sequences. LayerNorm normalizes across the feature dimension for each individual sample — fully independent of batch size and sequence length. This makes it suitable for autoregressive inference where batch_size=1. Modern LLMs (Llama, Mistral) use RMSNorm, which drops the mean-centering step: just divides by RMS. This is ~10-15% faster with equivalent quality in practice.

Cause: Attention heads in long contexts distribute attention less uniformly; positional biases from training. Mitigations: (1) Reranking — put most relevant chunks first and last, not middle. (2) Reduce context size — RAG with tight budgeting is better than stuffing everything in. (3) Use models specifically trained for long-context retrieval (e.g., Gemini 1.5 Flash with its multi-needle-in-haystack training). (4) Map-reduce: process sections independently, then synthesize. What is temperature, top-p, and top-k sampling? What are the practical differences?Junior›
Temperature (T): divides logits by T before softmax. T<1 → sharpens distribution (more deterministic). T>1 → flattens distribution (more creative/random). T=0 → greedy (always pick most likely token). Top-k: sample only from the k most probable tokens, regardless of probability mass. Top-p (nucleus): sample from the smallest set of tokens whose cumulative probability exceeds p (e.g., p=0.9). Adapts to the distribution — for sharp distributions, top-p might select just 5 tokens; for flat distributions, it selects many. For EPAM's production chatbots: temperature=0.1-0.3 for factual Q&A, temperature=0.7-1.0 for creative tasks. Top-p=0.9 is almost always the right choice over top-k (more adaptive).
How does LoRA reduce the number of trainable parameters? What is rank-r factorization?Senior›
LoRA (Hu et al., 2021): for a weight matrix W ∈ ℝ^(d×d), instead of updating W directly (d² params), we inject ΔW = BA where B ∈ ℝ^(d×r), A ∈ ℝ^(r×d), with r << d. For GPT-3 d=12288, full fine-tune = 150M params per layer. With r=8: 2×12288×8 = 196K params — a 750× reduction. The hypothesis: weight updates during fine-tuning have low intrinsic rank (they lie in a low-dimensional subspace). QLoRA adds: quantize base model to 4-bit NF4, keep LoRA adapters in fp16. This lets you fine-tune a 70B model on 2 A100 GPUs. Key interview point: LoRA adapters can be merged into base weights at inference with zero overhead: W' = W + BA.
What are hallucinations in LLMs, and what are the three classes of strategies to reduce them?Mid›

What are the 4 main failure modes of RAG systems, and how do you diagnose each?Senior›
(1) Retrieval Failure: Correct chunks exist but aren't retrieved. Diagnose: context_recall metric in RAGAS. Fix: improve embedding model, add hybrid retrieval, adjust chunk size. (2) Context Noise: Retrieved chunks are retrieved but irrelevant, diluting the context. Diagnose: context_precision metric. Fix: reranking, stricter similarity threshold. (3) Generation Failure: Correct chunks retrieved but LLM doesn't use them. Diagnose: faithfulness metric — answer not grounded in context. Fix: stronger system prompt ("answer ONLY from context"), better base model. (4) Chunking Failure: Information split across chunk boundaries, neither chunk is complete enough. Diagnose: manual inspection of retrieval results. Fix: parent-child chunking, larger chunk size with overlap. Most RAG debugging in production hits failure mode 1 or 4 first.

What is the GIL in Python, and how does it affect asyncio and multiprocessing?Senior›
GIL (Global Interpreter Lock): a mutex in CPython that allows only one thread to execute Python bytecode at a time. This makes CPython thread-safe but limits CPU parallelism. Asyncio: runs on a single thread, uses the event loop for cooperative multitasking. GIL doesn't matter because there's only one thread. I/O-bound tasks (HTTP calls, file reads) release the GIL while waiting — so asyncio concurrency is real for I/O. CPU-bound tasks: asyncio provides no speedup. GIL prevents true parallel execution. Solution: ProcessPoolExecutor (separate processes, each with own GIL) for CPU-bound work. Python 3.13 introduced "free-threaded" mode (PEP 703) that disables the GIL — still experimental but coming. For LLM systems: use asyncio for all API calls (I/O-bound), ProcessPoolExecutor for embedding computation (CPU-bound), never threading for CPU work.
Explain Python's descriptor protocol and how it powers property, classmethod, and staticmethod.Senior›
Descriptors: objects with __get__, __set__, __delete__ methods that define attribute access behavior. When you access obj.attr, Python calls type(obj).attr.__get__(obj, type(obj)) if attr is a descriptor. Data descriptors (__get__ + __set__): take priority over instance __dict__. Non-data descriptors (__get__ only): can be overridden by instance dict. property is a data descriptor — it intercepts both get and set. classmethod is a non-data descriptor — __get__ returns a bound method with the class (not instance) as first arg. staticmethod is a non-data descriptor — __get__ returns the function unchanged (no binding). This is why @property, @classmethod, and @staticmethod all work the same syntactic way. EPAM tests this for custom ORM-like systems or configuration classes. How does Python's memory management work? Explain reference counting and the garbage collector.Mid›
CPython uses two mechanisms: (1) Reference counting: every object has ob_refcnt. When refcnt reaches 0, memory is freed immediately. Fast and deterministic, but can't handle circular references. (2) Cyclic garbage collector (gc module): detects and collects reference cycles. Runs in generations (0, 1, 2) — gen 0 collected most frequently (young objects). gc.disable() for performance-critical loops where you manage lifetime manually. Memory leak patterns in LLM systems: circular references in conversation chains, keeping large embedding tensors in class attributes (use __slots__ to reduce per-object overhead), not closing async generators (must await .aclose()). Tools: tracemalloc (stdlib), memray, objgraph. For 10M document processing: use generators (O(1) memory), explicitly del large objects + gc.collect() after processing batches.
